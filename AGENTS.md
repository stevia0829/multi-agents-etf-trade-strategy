# AGENTS.md — 代码生成 / 修改约束

> 适用对象：所有为本项目（Eino Multi-Agent ETF Strategy）生成、修改、重构代码的自动化 / 半自动化 Agent。
> 作用：把"用户长期目标 + 项目既有事实"变成可执行的硬约束，让后续每一次代码改动都不偏离主线。
> **凡是与本文件冲突的"看起来更聪明的改动"——一律先拒绝，再问用户。**

---

## 0. 项目背景一句话

A 股场内 ETF 多 Agent 选股系统。**用户使用方式 = `advice` 模式每日开盘前出一份决策报告（提醒用），不做高频交易；目标 = 长期、稳定、可解释的超额收益**，不是博取短期峰值。所有改动都必须围绕这个目标。

---

## 1. 北极星指标（任何改动前先回答）

| 维度 | 优先级 | 说明 |
|---|---|---|
| **长期稳定性** | P0 | 长样本（≥1 年）回测 Sharpe / Calmar / 最大回撤 优先于 短样本累计收益 |
| **可解释性** | P0 | advice 模式输出的每一项（动作、止损、仓位）必须能追溯到具体因子 |
| **回撤控制** | P0 | 在 bear / risk_off 期不爆仓比"在 bull 期多赚"更重要 |
| **短期峰值收益** | P3 | 单月 / 单次回测的累计收益 **不是** 优化目标 |
| **代码复杂度** | P1 | 新增因子必须能讲清"在哪类行情起作用、在哪类行情副作用最大" |

> **判定规则**：如果一个改动只在 5 个月单边样本里 +5pp，但在 2 年长样本里 Sharpe 下降 / 最大回撤上升 ≥ 2pp，**视为负向**，必须拒绝或加 Regime 闸门。

---

## 2. 使用模式约束（重要）

- **主使用模式 = `advice`**：每天人工跑 `go run .`，看报告决定是否操作。
- **`backtest` 模式仅用于"验证因子是否长期有效"**，不是用户的实际交易方式。
- **任何"高频 / 日内 / 动态止损 / 移动止盈"机制**——除非用户明确要求，否则一律按"风险/收益不对称"原则拒绝。理由：
  1. advice 模式只在每天开盘前给一次信号，日内无法响应；
  2. A 股 ETF 单日 ±3% 是常态，移动止损在趋势市会反复触发，损失双边费率 + 错过反弹。
- **优先做"开盘前可执行的决策"**：动作、入场价、止损、止盈、仓位上限。**不要做需要日内监控的逻辑**。

---

## 3. 因子改造硬规则

### 3.1 任何新增 / 修改的"风险层因子"必须支持 **Regime-aware 闸门**

历史教训（写在 [docs/CHANGELOG-strategy.md](docs/CHANGELOG-strategy.md)）：
- **P1-2 凸性调整（÷σ_21）** 全局启用 → 在温和单边市把高动量赢家强行折价 → 短样本 -19pp。
- 修复方案：增加 `RegimeAwareP1` 开关（[agent/rotation.go](agent/rotation.go)），仅在 `bear / risk_off` 时生效。

**强制约束**：今后所有"基于波动率 / 下行风险 / 凸性 / 残差"的新因子，默认行为必须是
```
if Regime ∈ {bear, risk_off, ""(未知)} → 因子生效
else                                      → 因子跳过，回退到 P0 行为
```
不允许"无差别全局打折"赢家组。

### 3.2 仓位类机制必须分层

| 层级 | 允许做什么 | 不允许做什么 |
|---|---|---|
| **入场层（Screener / Rotation）** | 横截面排名、Regime 闸门、折溢价反向因子（已实现于 [agent/screener.go](agent/screener.go)） | 全局乘以 `1/σ`（量纲会暴力压制赢家） |
| **决策层（FinalAgent）** | 动作降档（buy → wait）、仓位上限（CapByPremium） | 触发"日常波动止损"（advice 模式无法执行） |
| **持仓层** | 仅"信号反转 / Regime 切换 / 灾难性止损（如 entry × 0.88）" | 移动止损 / 网格止盈 / 时间止损 |

### 3.3 引入新因子前的"自检三问"

1. 在哪类 Regime 起正向作用？在哪类 Regime 起反向作用？（必须能明确回答）
2. 与现有因子的相关性 / 量纲冲突？（特别注意"波动率类"因子两两叠加）
3. 长样本（≥1 年，至少跨一次 bear / 一次 bull）回测能否跑赢 P0？（短样本结果不算数）

**任何一问回答不出来 → 不要写代码，先回到用户对齐。**

## 4. 代码风格 / 仓库结构约束

### 4.1 包结构（不轻易改）

```
agent/        各路 Agent + Rotation/Screener/Final
backtest/     回测引擎（engine.go 是单文件单状态机，慎重改）
datasource/   数据源（东方财富 + 资金流 + 新闻 + 指数）
indicator/    动量 / 波动率 / 趋势指标（纯函数，无状态）
llm/          DeepSeek 客户端 + 降级链 + JSON 兜底
orchestrator/ Pipeline 编排
report/       Markdown 报告生成
types/        共享类型
```

**不允许**：
- 在 `indicator/` 下引入 IO / 网络（必须保持纯函数）
- 在 `agent/` 下直接读文件（统一走 `datasource/` 或 `report/`）
- 引入新的第三方依赖前必须先评估能否用标准库 / 现有依赖完成

### 4.2 注释 / 命名

- 所有"风险层因子"必须在函数 / 字段注释里写清：**机理来源（论文 or 经验）+ 适用 Regime + 失效场景**。参考 [agent/rotation.go](agent/rotation.go) `RegimeAwareP1` 字段的注释格式。
- 中文注释保持中文，不要中英夹写。
- 不写废话注释（如 `// 设置变量 a 为 1`）。

### 4.3 测试

- `indicator/` 下纯函数必须有单元测试。
- `agent/` 下涉及"Regime 闸门"或"评分修正"的逻辑，必须有覆盖至少 2 个 trend 分支的单测。
- 不要为了刷覆盖率写无意义测试。

---

## 5. 回测验证规范

### 5.1 不接受"短样本一锤定音"

- 默认验证窗口：**至少 1 年，跨一次 bear（如 2024-09 暴跌期）**。
- 如果只能跑短样本（数据缺失），必须在改动总结里**显式标注"短样本结论，长样本待验证"**。

### 5.2 必看指标（按优先级）

1. 累计收益 / 年化（看有没有比 buy & hold 强）
2. **最大回撤 + 最大回撤恢复天数**（advice 用户最关心）
3. **Sharpe / Calmar**（稳定性）
4. 实际建仓数（避免"高频换手吃费率"）
5. 胜率（参考即可，不作为主指标）

### 5.3 A/B 对比报告（已有）

`--bt-variant=both_p1` 已经支持 P0 vs P1 对比（[main.go](main.go) `runBacktest`）。**新加因子必须复用这套对比**，不要为了出"漂亮数字"换样本区间。

---

## 6. advice 模式特别约束

- 报告里给的 `止损 / 止盈` 必须经过校验（参考 P2-4 picks 数值校验思路）：
  - `stop ∈ [price × 0.88, price × 0.97]`
  - `take ∈ [price × 1.05, price × 1.20]`
  - `stop < price < take`
- 不要在 advice 报告中加"日内动态调整"建议（用户做不到）。
- 不要在 advice 报告中给"短线 1 日内必赚"承诺（不符合长期稳定收益的目标）。
- 持仓对照（[agent/common.go](agent/common.go) `BuildHoldAdvice`）保持现状："信号还在 → 继续持有"是首要原则，避免让用户高频换手。

---

## 7. 提示给后续 Agent 的"红线清单"

> 以下行为**必须先和用户对齐**，不允许擅自实施：

1. ❌ 任何"全局乘以 1/σ"或"vol-targeting 全局版"
2. ❌ 任何会触发频繁换手的机制（移动止损、ATR 跟踪、N 日不涨清仓 等）
3. ❌ 删除 `Regime-aware` 闸门，把因子改回"全局启用"
4. ❌ 把回测结论建立在 < 6 个月的样本上
5. ❌ 引入新的 LLM 调用 / 新的 API（每次新增都增加运行成本与失败面）
6. ❌ 把 `advice` 模式输出改成需要日内执行的指令
7. ❌ 在 `indicator/` 下引入 IO / 状态
8. ❌ 为了"看起来全面"把单文件拆分成多文件（小项目优先单文件可读性）

> 以下行为**鼓励主动实施**：

1. ✅ 在新因子注释里写清"适用 Regime / 失效场景"
2. ✅ 引入新风险层因子时同步加 `RegimeAware*` 开关
3. ✅ 长样本回测验证 + 必看指标全量给出
4. ✅ 把 LLM 输出做 sanitization（picks 数值校验等）
5. ✅ 用现有 `both_p1` 模板做 A/B 对比，而不是另起炉灶

## 8. 改动提交模板（建议自查）

每次改动结束前，回答以下问题（写进 [docs/CHANGELOG-strategy.md](docs/CHANGELOG-strategy.md)）：

```
## 改动 ID / 日期
- 动机：（用户长期目标的哪个维度？）
- 机理：（论文 / 经验 / 数据观察）
- 适用 Regime：（bull / neutral / bear / risk_off）
- 失效场景：（什么时候会有反向作用？已加什么闸门？）
- 验证窗口：（起止日期 + 是否跨 bear）
- 必看指标对比：累计 / 回撤 / Sharpe / 建仓数 / 胜率
- 风险点：（已知的 trade-off）
- 回滚方式：（关闭哪个开关 / 还原哪一行）
```

---

**最终原则**：当用户没明确表态时，**永远默认"长期稳定 + 可解释 + 不增换手"**。短期收益的诱惑是后续 Agent 最大的偏差源，必须由本文件兜底。
